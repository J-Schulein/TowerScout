TowerScout V1 RC1

Project Overview

TowerScout is a local web application for identifying likely cooling towers from satellite and aerial imagery for outbreak investigation and registry-building workflows.

What The V1 RC1 Package Provides

The V1 RC1 package provides a Windows-first local pilot path built around a GitHub Release control ZIP, a pinned GHCR container image digest, and a separately supplied asset bundle.

Runtime Docker-compatible or OCI-compatible Compose runtime configuration.
Scripts Windows launch, stop, status, log, asset import, and TLS CA helper scripts.
Docs Package-local setup, support, workflow, and overview docs.
Notices Source, license, provider, model, data, SBOM, and release-manifest files.

The normal pilot path is package-based. Source checkout, Python virtual environment, and Conda setup guides are legacy/source-install support material, not the preferred V1 RC1 pilot path.

Main Workflow

  1. Start TowerScout from start.bat to initialize the package.
  2. Import required model and ZIP-code assets.
  3. Configure Google Maps or Azure Maps in Setup Wizard or Settings.
  4. Choose a provider, define a search area, and estimate tile count.
  5. Run detection, review likely towers, and add manual corrections if needed.
  6. Export CSV, KML, or dataset results according to site policy.

The full workflow guide is available at TowerScout User Guide.

Release Boundary

The V1 RC1 supported target is Windows 11 AMD64, single-user local use, CPU baseline, normal outbound internet access, a Docker-compatible or OCI-compatible runtime with Compose support, and one site/user-owned restricted Google Maps or Azure Maps provider key.

Out of scope for V1 RC1: macOS, ARM64, air-gapped/offline installs, VDI, shared multi-user hosting, managed remote deployment, and native installer behavior.

Provider Keys

TowerScout uses Google Maps or Azure Maps browser SDKs. Browser map SDK keys are visible to someone with access to the running browser app.

  • V1 RC1 assumes site/user-owned restricted provider keys.
  • Unrestricted shared TowerScout project keys are unsupported.
  • Sites should configure provider restrictions, API/service scoping, quotas, billing alerts, monitoring, and key rotation according to local policy.

License And Source

The YOLO-enabled V1 RC1 package/image is not Apache-2.0-only. It is distributed with AGPL-3.0 obligations because it includes Ultralytics YOLOv5 runtime source and YOLO-derived detector weights.

The release package includes LICENSE, NOTICE, THIRD_PARTY_NOTICES.md, MODEL_LICENSES.md, DATA_LICENSES.md, PROVIDER_TERMS.md, SOURCE.txt, SBOM.txt, and release-manifest.v1.json.

When TowerScout is running, the local source/license notice is available at /license.
